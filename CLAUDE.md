# CLAUDE.md — growing-brand (contributor side)

## What this is

An ambient brand-memory layer for AI-assisted design work. Audience = designers and design-engineers using AI coding tools. Assume users have never seen this repo's internals and never will — the tool has to work inside their normal flow.

## Audience rules (the reason this tool is shaped the way it is)

- Users will **not** do daily practice, review sessions, or any routine. If a feature needs discipline, it's wrong.
- Users will **not** run judging ceremonies (generate K options, rate them, write reasons). Their judgments already happen inside normal work — *"not this one", "warmer", "approved"* — the tool's only job is to catch them there.
- Capture is **ambient and silent**; distillation is **automatic** (threshold-triggered, no permission asked) but **visible in ≤3 lines** and reversible — a pushback on a promotion is itself a judgment.
- Plain language everywhere. Public files (README, SKILL.md, templates) use only: core / tokens / moves / journal.

## Design principles

- **Zero runtime dependencies.** The whole tool is markdown: two skills + four templates. No scripts, no node_modules, no server. The AI harness is the runtime. Keep it that way — any feature that needs code should justify itself hard.
- **Files are the product.** The grown `brand/` folder is the value; the skills are just the gardener. Files must stay readable, diffable, portable, and edit-friendly (hand edits = strongest signal).
- **Provenance always.** No rule enters core/tokens/moves without citing journal entries. A rule that can't show its evidence doesn't get written.
- **Contradictions stay open.** Context-split first; otherwise Tensions. Never resolve by fiat; demotion exists.
- **Core stays small.** ~15 rules max. When in doubt, it's a token or a move.
- **The user's words beat paraphrase.** Quote them in the journal.

## Layout

```
.claude-plugin/plugin.json      manifest (validated with `claude plugin validate .`)
.claude-plugin/marketplace.json makes the repo directly installable as a plugin
skills/brand-memory/SKILL.md    the ambient layer: apply → capture → distill (the heart)
skills/brand-init/SKILL.md      one-time onboarding: scan + ≤3 questions + seed
skills/brand-init/templates/    core.md · tokens.md · moves.md · journal.md
example/                        fictional brand (Ridgeline) — generated by executing the
                                loop end-to-end once; regenerate it the same way if the
                                spec changes, don't hand-tweak it out of sync
```

## How to test a change

There is no test suite — the spec *is* the program and the AI is the runtime. So test by **dry run**: create a sandbox project with some CSS, execute `/brand-init` faithfully (real greps, real files), simulate ~12 judgments across ≥3 fake jobs, let the distill trigger fire, and check: entry numbering, marker update, provenance on every promoted line, observed/seeded upgrades, [stated] fast-path, tension handling. Any ambiguity you hit while executing is a bug in the SKILL text — fix the text, not the run.
